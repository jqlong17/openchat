# OpenChat

仓库：<https://github.com/jqlong17/openchat>

## 项目定位

**产品目标**：开发一款**类似微信**的即时通讯与社交产品，并**支持接入 [OpenClaw](https://github.com/openclaw/openclaw)**，使用户能在熟悉的聊天场景里与 Agent 协作。

**核心价值**：**开源、开放**——代码与协作方式可审计、可 fork、可私有部署；这是我们与封闭「围墙花园」产品的根本区别。

**技术栈与生态**

- **语言**：**服务端技术栈统一为 Rust**（IM 核心、API、集成层均以 Rust 实现），兼顾性能、并发与可分发性（如单二进制部署）；OpenClaw 等为独立系统，通过 HTTP 对接。
- **参考**：工程与产品形态上**参考 [VoceChat](https://github.com/Privoce/vocechat-server-rust)** 等开源项目（轻量、易集成、私有化友好）。
- **生态**：目标是成为 **OpenClaw 开放生态中的一环**——在消息与连接器层面与 OpenClaw 对齐，而不是再造一套封闭的 IM 孤岛。

更细的调研与契约设计见 `workflow/`。

**MVP 约定（当前）**：优先交付 **HTTP API + 自动化测试 + 命令行验收**；与 OpenClaw 的对接以 **协议打通 + Mock** 为里程碑，真机联调不阻塞 MVP。

## 我们为什么做 OpenChat

### 行业正在进入下一阶段

**1. 从「会聊天的机器人」到可执行的智能体（Agent）**  

以 [OpenClaw](https://github.com/openclaw/openclaw) 为代表的一类项目，带动了整个行业：AI 不再停留在对话式 Chatbot，而是快速转向 **可连接外部系统、能执行任务** 的 **Agent（智能体）** 形态。与此同时，原本彼此孤立的软件边界被冲淡——用户已经可以在 **Telegram、Discord** 等通讯工具里，通过 AI 去操作本地或云上的计算机与服务器。人机协作的入口，正在从「单一 App」扩散到「对话与频道」。

**2. 模型能力与授权边界同步上移**  

以 Claude、GPT 等新一代模型为例，幻觉显著降低，系统与工具权限在可控前提下被逐步放开：AI 可以驱动终端、生成更可审计的代码，并进一步触达真实世界——从业务系统到摄像头、机器人等物理设备。这些变化共同说明：**AI 正在进入「能持续作用于环境」的下一阶段**，而不仅是内容生成。

### 国内生态里我们看到的结构性问题

**1. 「类 Claw」很多，开放却很少**  

腾讯、阿里、Kimi、MiniMax 等厂商都在推出各自的「Claw 系」能力，但多数 **既不开放协议、也不开放实现**——这与 OpenClaw 最核心的 **开源与可审计** 形成反差；若智能体只能锁在单一厂商体系内，生态很难真正生长。

**2. 超级 App 仍在加固围墙**  

以微信等为代表的产品，仍在各自体系内堆叠能力，用户不得不在 **多个封闭花园** 之间切换。若长期如此，人与 AI 的协作会被 **割裂的产品体验** 绑死，这与「Agent 跨系统行动」的下一阶段 **是相悖的**。
