# 05 协作与演进治理（设计稿）

**目的**：在 [01-技术架构](./01-技术架构.md)、[03-API与集成契约](./03-API与集成契约.md)、[04-代码仓库结构](./04-代码仓库结构.md) 之上，约定 **多角色协作、契约演进、运维边界**，降低「口头对齐」带来的返工。  
**适用**：贡献者增多、拆分 crate、或与 OpenClaw 并行迭代时启用；小团队可先只做其中几条。

---

## 1. 契约的单一事实来源

| 层级 | 约定 |
|------|------|
| **对外 HTTP** | 以仓库内 **`openapi.yaml`**（或 `openapi/` 分文件）为**机器可读权威**；CI 中校验「实现与 OpenAPI 一致」或「OpenAPI 由代码注解生成」二选一，并在 `CONTRIBUTING.md` 写死。 |
| **Markdown（03 等）** | 用于人类可读说明、示例与背景；若与 OpenAPI 冲突，**以 OpenAPI 为准**，并修订 Markdown。 |
| **WebSocket** | 事件类型与字段在 OpenAPI 的 `components` 扩展或单独 **`websocket-events.md`** 中维护，避免只写在散落 PR 描述里。 |

---

## 2. 代码所有权（CODEOWNERS）

在仓库根增加 **`.github/CODEOWNERS`**（路径以实际 crate 为准），例如：

```text
# 示例：按 crate 分配审查责任（账号名替换为真实团队/个人）
/crates/openchat-core/       @team-core
/crates/openchat-server/       @team-server
/crates/openchat-integration/ @team-integration
/workflow/02-design/          @team-arch
```

**原则**：触及 `openchat-core` 或 **`openapi.yaml` 的 breaking change** 时，必须经对应 owner 或架构评审（见第 4 节）。

实现阶段若仍为单 crate，可按 `src/routes/`、`src/domain/` 等路径划分 owner。

---

## 3. 架构决策记录（ADR）

重大且**难逆转**的决策（换存储、消息 ID 策略、多租户模型、API 主版本升级）用 **ADR** 留痕。

**存放**：`workflow/02-design/adr/` 或 `docs/adr/`，文件名：`0001-use-sqlite-for-mvp.md`。

**模板（可直接复制）**：

```markdown
# ADR-0001：<标题>

## 状态
提议 | 已接受 | 已废弃（被 ADR-00xx 取代）

## 背景
<问题与约束>

## 决策
<我们决定做什么>

## 后果
<正面 / 负面 / 后续工作>
```

---

## 4. API 与数据模型的变更流程

| 变更类型 | 建议流程 |
|----------|----------|
| **非破坏**（新增可选字段、新端点） | PR 说明 + 更新 OpenAPI + 必要时更新 `03`。 |
| **破坏**（删字段、改语义、改 URL） | 简短 **RFC**（可仅为 `workflow/` 下一页或 Discussion）：动机、迁移方案、版本号；至少一名 **core/api** owner 认可后再合入。 |
| **数据模型** | [02-数据模型](./02-数据模型.md) 与迁移脚本**同一 PR** 或强绑定，避免「先合代码后补文档」。 |

---

## 5. 与 OpenClaw 的协作边界

| 项 | 约定 |
|----|------|
| **契约** | 出站负载 `schema`（如 `openchat.outbound.v1`）变更时，**同时**更新 [03](./03-API与集成契约.md) 与 OpenClaw 侧连接器 README（若在别仓，用链接或子模块说明）。 |
| **版本** | 主版本 bump 时列出 **OpenClaw 最低兼容版本**（若可测）；或在发布说明中写「需同步升级连接器」。 |
| **责任** | 连接器默认由 **OpenChat 集成域** 或 **OpenClaw 社区扩展** 维护，在各自 README 写明；避免无 owner 的「中间协议」。 |

---

## 6. 可观测性与运维（原则）

不替代具体选型，仅定**必须覆盖的维度**，便于扩展与排障：

| 维度 | 说明 |
|------|------|
| **日志** | 结构化（含 `request_id`、`conversation_id` 可选）；禁止长期打印完整消息正文到 info（隐私）。 |
| **指标** | 至少：请求延迟、错误率、WS 连接数、出站 Webhook 失败率。 |
| **追踪** | 跨 OpenChat → OpenClaw 的请求带 **`trace_id`** / `event_id`（与 [02](./02-数据模型.md) `integration_meta` 对齐）。 |

详细部署与 SLO 可在后续 **`06-运维与SLO.md`** 或 ADR 中展开。

---

## 7. 安全与密钥

- **密钥、生产配置**不进 Git；`.env.example` 仅变量名。  
- **Webhook HMAC**、API Key 轮换策略在 ADR 或安全小节中**有记录**，而非只存在私聊。  
- 重大安全修复：CHANGELOG / 安全公告按项目策略发布。
